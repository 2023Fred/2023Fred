§ 01 / 06
O contexto

Um laboratório deliberadamente imperfeito.

Este relatório documenta uma sessão real de administração Oracle conduzida em um ambiente controlado — uma máquina virtual Oracle Linux 8 rodando três instâncias de banco em paralelo, configurada para fins exclusivamente didáticos.

Aviso importante
Trata-se de um laboratório de testes controlado. Todas as vulnerabilidades de segurança, alertas de capacidade e configurações fora do padrão foram deliberadamente deixadas sem hardening para servir de base aos exercícios práticos. Nenhum dado real ou sensível existe neste ambiente. Os achados aqui descritos não representam falhas operacionais, e sim oportunidades pedagógicas mapeadas pelas skills de auditoria.

ACID — os quatro pilares que governam o ambiente

O laboratório foi montado para demonstrar o cumprimento das quatro propriedades fundamentais que todo banco de dados transacional precisa garantir. Em Oracle, essas propriedades não são "boas práticas" — são contratos que o motor honra em cada commit. Compreendê-las é o ponto de partida para qualquer conversa sobre confiabilidade de dados.

A
Atomicidade
Uma transação acontece por inteiro ou não acontece. Não existe meio caminho. Garantida pelos undo segments do Oracle, que permitem desfazer qualquer operação interrompida antes do commit.
C
Consistência
O banco passa sempre de um estado válido para outro estado válido. Constraints, triggers, foreign keys e regras de integridade são o aparato que o Oracle aplica em cada operação DML.
I
Isolamento
Transações concorrentes não enxergam dados sujos umas das outras. O Oracle implementa via MVCC (Multi-Version Concurrency Control), servindo leituras consistentes a partir de undo blocks.
D
Durabilidade
Uma vez confirmada, a transação sobrevive a qualquer falha — energia, crash, desastre. Os redo logs e o archivelog mode são a espinha dorsal dessa garantia em Oracle.

§ 03 / 06
A topologia

Três instâncias, um único servidor.

O laboratório roda em um host Oracle Linux 8 com 3,5 GB de RAM e 2 cores Intel, hospedando simultaneamente um banco primário, uma duplicata RMAN e um standby físico Data Guard. É a estrutura mínima para praticar cenários realistas de alta disponibilidade.

§ 04 / 06
A metodologia

Nove skills, uma orquestradora.

Cada área crítica do trabalho de um DBA Oracle foi encapsulada em uma skill — um módulo especializado com prompts, queries e critérios de diagnóstico próprios. A nona skill, central-agentes, é o maestro: consolida os achados das outras oito em um relatório executivo único.

01

check-oracle-health

Health check completo: tablespaces, sessões, wait events, SGA/PGA, jobs do scheduler e alert log.

▲ FRA em 79%, backups com falha
02

gerenciar-tablespaces

Criação, redimensionamento e gestão de tablespaces — permanent, temp, undo e bigfile.

● Espaço saudável, UNDO ajustável
03

gerenciar-redo-logs

Dimensionamento, multiplexação e redimensionamento de online redo logs.

▼ Sem multiplexing — risco alto
04

criar-pdbs

Provisionamento e administração de Pluggable Databases em arquitetura Multitenant.

▲ PDB sem SAVE STATE
05

configurar-hugepages

Configuração de HugePages no Linux para otimização de memória SGA do Oracle.

▼ Não configurado (4K pages)
06

auditoria-seguranca

Hardening, Unified Auditing, TDE, perfis de senha e revisão de privilégios excessivos.

▼ Senha default, PUBLIC exposto
07

capacity-planning

Projeção de crescimento de tablespaces, FRA, sessões e CPU via dados de AWR.

▲ FRA esgota em ~11 dias
08

licenciamento-oracle

Auditoria de edition, options pagas e features ativas via DBA_FEATURE_USAGE_STATISTICS.

● 1 licença Processor · revisar packs
09 · ORCHESTRATOR

central-agentes

Orquestrador multi-agente. Consolida o output das oito skills em um plano executivo priorizado.

★ Sintetiza · prioriza · entrega
